## 分支模型

首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；
那在哪干活呢？干活都在dev分支上，也就是说，dev分支是不稳定的，到某个时候，比如1.0版本发布时，再把dev分支合并到master上，在master分支发布1.0版本；
你和你的小伙伴们每个人都在dev分支上干活，每个人都有自己的分支，时不时地往dev分支上合并就可以了。
新的分支创建后，我们可以对其中的文件进行新增、修改、删除等操作，并且这些变更不会影响到其它分支。

## 创建分支
使用"git branch"命令可以管理分支，"git checkout"命令则能够切换分支。
首先创建一个分支"dev"，并切换到此分支。
```bash
# 创建分支"dev"
git branch dev
# 切换至分支"dev"
git checkout dev
Switched to branch 'dev'
# 查看当前项目拥有的分支
git branch
* dev
  master
```
"git branch"命令不添加任何参数时，可以列出项目的所有分支，当前所在分支前会标记为"*"。
使用"git checkout -b <分支名称>"命令可以创建分支并立刻切换到新分支，用于简化操作。
由于"git checkout"命令能操作文件与分支，容易使人迷惑，较新的Git版本提供了"git switch"命令用于切换分支，"git switch -c <分支名称>"能够创建并切换至新的分支，"git switch <分支名称>"则能够切换至已存在的分支。
我们在新分支上进行的修改与提交，并不会影响到其它分支，可以相对独立地进行工作。
```bash
# 查看"2.txt"原始内容
zqg@zqg-OptiPlex-3070:~/test$ cat 2.txt 
1234
# 清空原内容，写入新内容。
zqg@zqg-OptiPlex-3070:~/test$ echo "abcd" > 2.txt 
zqg@zqg-OptiPlex-3070:~/test$ git add 。
zqg@zqg-OptiPlex-3070:~/test$ git commit -m "修改2.txt-dev"
[dev 5885102] 新分支-初始提交
 1 file changed, 1 insertion(+), 1 deletion(-)
# 切换至主要分支，查看"2.txt"的内容，主分支中的文件并没有被更改。
zqg@zqg-OptiPlex-3070:~/test$ git checkout master 
切换到分支 'master'
您的分支与上游分支 'origin/master' 一致。
zqg@zqg-OptiPlex-3070:~/test$ cat 2.txt 
1234
```

## 合并分支
当我们在新创建的分支上完成开发任务后，就可以将变更合并至主要分支，并移除旧的分支。
"git merge"命令用于将指定的分支合并至当前所在分支，我们需要将"dev"分支的变更合并至"master"分支，因此首先切换至"master"分支，然后再执行合并。
```bash
# 切换至"master"分支
zqg@zqg-OptiPlex-3070:~/test$ git checkout master 
已经位于 'master'
您的分支与上游分支 'origin/master' 一致。
# 合并"dev"分支的变化
zqg@zqg-OptiPlex-3070:~/test$ git merge dev
更新 dcff536..5885102
Fast-forward
 2.txt | 2 +-
 1 file changed, 1 insertion(+), 1 deletion(-)
zqg@zqg-OptiPlex-3070:~/test$ git log --graph --pretty=oneline
* 781c0cb17ba210f01c24b523e2ea75fc6dc59495 修改1.txt-dev
* 907b9b25fd9ae4a9d17dff68ca3027564436c29c 初始化
```
因为"master"分支没有发生变化，"dev"分支在其基础上进行修改，因此不存在冲突，Git使用"Fast-forward"（快进）模式进行合并，直接将"master"指向"dev"的当前版本，完成合并。
此时合并任务已经完成，"dev"分支中的变更已经写入"master"分支，可以将"dev"分支删除。
```bash
# 查看"master"分支上的"2.txt"
zqg@zqg-OptiPlex-3070:~/test$ cat 2.txt 
abcd
# 删除"dev"分支
zqg@zqg-OptiPlex-3070:~/test$ git branch -d dev
已删除分支 dev（曾为 5885102）。
# 查看项目的所有分支
zqg@zqg-OptiPlex-3070:~/test$ git branch 
* master
```

## 解决合并冲突
如果同一个文件在不同分支中均被修改，或者团队协作时不同成员修改了同一个文件，进行合并时就会产生冲突，Git不知道哪一方提交的文件是我们最终期望的，此时就需要我们手动编辑文件，解决冲突。
首先创建一个空的项目，并在文本文件中写入一些内容。
```bash
zqg@zqg-OptiPlex-3070:~$ mkdir Test
zqg@zqg-OptiPlex-3070:~$ cd Test/
zqg@zqg-OptiPlex-3070:~/Test$ git init
初始化空的 Git 仓库于 /home/zqg/Test/.git/
zqg@zqg-OptiPlex-3070:~/Test$ echo "Merge Test" > 1.txt
zqg@zqg-OptiPlex-3070:~/Test$ git add .
zqg@zqg-OptiPlex-3070:~/Test$ git commit -m "初始化"
[master （根提交） 2eb27e9] 初始化
 1 file changed, 1 insertion(+)
 create mode 100644 1.txt
```
创建一个分支"dev"，并修改"1.txt"的内容。
```bash
zqg@zqg-OptiPlex-3070:~/Test$ git checkout -b dev
切换到一个新分支 'dev'
zqg@zqg-OptiPlex-3070:~/Test$ echo "Merge Test dev" > 1.txt
zqg@zqg-OptiPlex-3070:~/Test$ git add .
zqg@zqg-OptiPlex-3070:~/Test$ git commit -m "修改文本-dev"
[dev 2e9cd23] 修改文本-dev
 1 file changed, 1 insertion(+), 1 deletion(-)
```
切换到主要分支，并修改"1.txt"的内容。
```bash
zqg@zqg-OptiPlex-3070:~/Test$ git checkout master 
切换到分支 'master'
zqg@zqg-OptiPlex-3070:~/Test$ echo "Merge Test master" > 1.txt
zqg@zqg-OptiPlex-3070:~/Test$ git add .
zqg@zqg-OptiPlex-3070:~/Test$ git commit -m "修改文本-master"
[master c6ba435] 修改文本-master
 1 file changed, 1 insertion(+), 1 deletion(-)
```
此时将"dev"分支合并至主要分支，就会产生冲突，导致合并失败，需要用户手动处理冲突的文件。
```bash
zqg@zqg-OptiPlex-3070:~/Test$ git merge dev
自动合并 1.txt
冲突（内容）：合并冲突于 1.txt
自动合并失败，修正冲突然后提交修正的结果。
zqg@zqg-OptiPlex-3070:~/Test$ git status 
位于分支 master
您有尚未合并的路径。
  （解决冲突并运行 "git commit"）

未合并的路径：
  （使用 "git add <文件>..." 标记解决方案）

	双方修改：   1.txt

修改尚未加入提交（使用 "git add" 和/或 "git commit -a"）
zqg@zqg-OptiPlex-3070:~/Test$ git checkout dev
1.txt: needs merge
error: 您需要先解决当前索引的冲突
```
我们可以使用"git status"命令查看冲突的文件，两个分支中的差异也已经写入文件中，用户根据实际需要修改文件后，进行一次提交，就可以再次尝试合并。
```bash
# 查看冲突的文件
zqg@zqg-OptiPlex-3070:~/Test$ git status 
位于分支 master
您有尚未合并的路径。
  （解决冲突并运行 "git commit"）

未合并的路径：
  （使用 "git add <文件>..." 标记解决方案）

	双方修改：   1.txt

修改尚未加入提交（使用 "git add" 和/或 "git commit -a"）
# 查看冲突的内容
zqg@zqg-OptiPlex-3070:~/Test$ cat 1.txt 
 <<<<<<< HEAD
 Merge Test master
 =======
 Merge Test dev
 >>>>>>> dev
# 手动修改文件至预期结果，并进行提交。
zqg@zqg-OptiPlex-3070:~/Test$ echo "Merge Test final" > 1.txt
zqg@zqg-OptiPlex-3070:~/Test$ git add .
zqg@zqg-OptiPlex-3070:~/Test$ git commit -m "解决1.txt冲突"
[master 55d9d31] 解决1.txt冲突
# 再次进行合并
zqg@zqg-OptiPlex-3070:~/Test$ git merge dev
Already up-to-date.
# 查看工作日志
zqg@zqg-OptiPlex-3070:~/Test$ git log --oneline
55d9d31 解决1.txt冲突
c6ba435 修改文本-master
2e9cd23 修改文本-dev
2eb27e9 初始化
```
手动修改冲突文件后，可以成功进行合并，此时日志中记录了两个分支的提交记录与解决冲突时的提交记录。

## 强制删除分支
如果希望放弃某个分支的更改，但此分支已经有提交记录，系统会阻止删除操作，此时使用"-D"选项可以强制删除分支。
```bash
# 创建新的分支，并进行提交操作。
zqg@zqg-OptiPlex-3070:~/test$ git checkout -b dev
切换到一个新分支 'dev'
zqg@zqg-OptiPlex-3070:~/test$ touch 2.txt
zqg@zqg-OptiPlex-3070:~/test$ git add .
zqg@zqg-OptiPlex-3070:~/test$ git commit -m "新增2.txt"
[dev 3bf32ef] 新增2.txt
 1 file changed, 0 insertions(+), 0 deletions(-)
 create mode 100644 myfile
zqg@zqg-OptiPlex-3070:~/test$ git checkout master 
切换到分支 'master'
# "-d"选项不能删除此分支
zqg@zqg-OptiPlex-3070:~/test$ git branch -d dev
error: 分支 'dev' 没有完全合并。
如果您确认要删除它，执行 'git branch -D dev'。
# "-D"选项可以强制删除此分支
zqg@zqg-OptiPlex-3070:~/test$ git branch -D dev 
已删除分支 dev（曾为 3bf32ef）。
```

## 禁用"Fast-forward"模式
Git在使用Fast Forward模式进行合并时，没有创建新的版本，只是把当前分支指向合入分支的最新版本，日志中不会体现合并操作，且不利于版本回退。我们可以在合并时添加"--no-ff"选项禁用该模式，使日志内容更详细。
```bash
# 创建新分支"dev"
zqg@zqg-OptiPlex-3070:~/test$ git checkout -b dev
切换到一个新分支 'dev'
# 创建新文件，并提交变更。
zqg@zqg-OptiPlex-3070:~/test$ touch 2.txt
zqg@zqg-OptiPlex-3070:~/test$ git add .
zqg@zqg-OptiPlex-3070:~/test$ git commit -m "新增2.txt-dev"
[dev 8f04ca6] 新增2.txt-dev
 1 file changed, 0 insertions(+), 0 deletions(-)
 create mode 100644 2.txt
# 将"dev"分支合并至主要分支
zqg@zqg-OptiPlex-3070:~/test$ git checkout master 
切换到分支 'master'
zqg@zqg-OptiPlex-3070:~/test$ git merge dev --no-ff -m "合并dev分支"
Merge made by the 'recursive' strategy.
 2.txt | 0
 1 file changed, 0 insertions(+), 0 deletions(-)
 create mode 100644 2.txt
# 查看Git日志
zqg@zqg-OptiPlex-3070:~/test$ git log --graph --pretty=oneline
*   4a8b4f2f7b190cd64914ea718838c1dde436e77b 合并dev分支
|\  
| * 8f04ca6bf54f45d3db3c478a9ab81b243e20c68e 新增2.txt-dev
|/  
* 907b9b25fd9ae4a9d17dff68ca3027564436c29c 初始化
```
使用"--no-ff"选项后会生成新的版本，因此也要加上"-m"选项填写日志内容，此时Git日志中体现了完整的分支合并历史。
如果我们希望每次合并时自动添加"--no-ff"选项，可以修改Git配置实现。
```bash
# 仅针对当前分支生效
zqg@zqg-OptiPlex-3070:git config branch.master.mergeoptions  "--no-ff"
# 仅针对当前版本库生效
zqg@zqg-OptiPlex-3070:git config --add merge.ff false
# 针对所有版本库生效
zqg@zqg-OptiPlex-3070:git config --global --add merge.ff false
``` -->
