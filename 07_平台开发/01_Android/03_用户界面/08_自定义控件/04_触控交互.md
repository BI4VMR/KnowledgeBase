# 简介
Android SDK内置的按钮、列表等控件提供了点击、滑动翻页等高级事件接口，我们通过注册监听器能够实现大多数通用的交互场景；对于一些特殊的交互场景，例如：绘图板、多层滑动控件嵌套等，内置控件无法满足需求，我们需要自行处理底层的交互事件，实现相关功能。

# 交互事件
在Android系统中，MotionEvent类用于描述用户与界面交互时产生的各种事件，例如：手指接触屏幕、手指在屏幕上滑动、鼠标单击等。

我们可以调用View的 `setOnTouchListener(OnTouchListener l)` 方法监听并处理触控事件，OnTouchListener接口的唯一方法 `boolean onTouch(View v, MotionEvent event)` 将在控件收到事件时触发，此处我们输出消息查看事件详情：


MotionEvent实例的 `int getAction()` 方法用于获取事件类型，常见的事件类型如下文列表所示：

- `ACTION_DOWN (0)` : 手指开始接触屏幕时触发一次该事件，它是一次触控交互的起点。
- `ACTION_UP (1)` : 手指离开屏幕时触发一次该事件，它是一次触控交互的终点。
- `ACTION_MOVE (2)` : 手指在屏幕上滑动时，将会反复触发该事件，其中包含手指在屏幕上的最新坐标。
- `ACTION_CANCEL (3)` : 后续事件被上层容器拦截时，触发一次该事件，通知当前控件后续操作应当取消。
- `ACTION_OUTSIDE (4)` : 手指触摸控件外部区域时触发一次该事件。控件默认接收不到该事件，我们需要在控件所在Window中配置 `FLAG_WATCH_OUTSIDE_TOUCH` 以启用功能。用户点击弹窗外部区域可以关闭弹窗就是通过该事件实现的。

一次触控交互流程包含多个事件，我们将其称为事件序列，常见的事件序列如下文列表所示：

- 点击：手指按住屏幕(ACTION_DOWN) -> 手指离开屏幕(ACTION_UP)
- 滑动：手指按住屏幕(ACTION_DOWN) -> 手指移动(ACTION_MOVE) -> 手指移动(ACTION_MOVE) -> ... -> 手指离开屏幕(ACTION_UP)

除了事件类型之外，MotionEvent实例还提供了一些额外的信息：

- `float getX()` : 获取事件触发位置相对所属控件的X轴坐标。
- `float getY()` : 获取事件触发位置相对所属控件的Y轴坐标。
- `float getRawX()` : 获取事件触发位置相对整个屏幕的X轴坐标。
- `float getRawY()` : 获取事件触发位置相对整个屏幕的Y轴坐标。
- `long getEventTime()` : 获取当前事件发生时刻的时间戳。
- `long getDownTime()` : 获取当前事件序列中 `ACTION_DOWN` 事件的时间戳。


# 交互流程
用户界面通常由Activity（或Window的根容器）、ViewGroup、View三个层级组成，当用户触摸屏幕上的某个位置时，Framework首先将事件传递给Activity，Activity寻找触摸位置最顶层的ViewGroup并下发事件，ViewGroup则继续寻找触摸位置最顶层的View并下发事件，View作为最终接收者可以对事件作出响应，执行预设逻辑或开发者通过监听器设置的逻辑。

触控事件的交互流程可概括为：递归下发（由外向内寻找接收者）与逐层响应（内层接收者消费事件则流程结束，无法响应事件则逐层请求外层容器处理。）。

当我们制作控件时，可以在以下两个环节插入自定义逻辑：

1. 自定义容器下发逻辑：我们可以按需拦截部分事件并自行处理，不再请求子View处理，例如：禁止ViewPager滑动、控制多层RecyclerView嵌套滑动。
2. 自定义控件处理逻辑：我们可以自定义控件对事件的响应，例如：在用户触摸位置绘制线段，实现绘图板功能。

触控事件的交互流程主要涉及以下方法：

🔷 `boolean dispatchTouchEvent(Event ev)`

该方法是Activity、ViewGroup、View所共有的，它是每个控件处理事件的入口，用于控制事件分发逻辑。

外层容器分发事件给内层控件时，将会逐层调用该方法：Activity调用ViewGroup的 `dispatchTouchEvent()` 方法，ViewGroup调用View的 `dispatchTouchEvent()` 方法。

若返回值为 `true` ，表示当前控件已经将该事件处理完毕，我们也称作事件被“消费”，整个处理流程结束。

若返回值为 `false` ，表示当前控件无法响应该事件，请求上层容器作出响应。

🔷 `boolean onInterceptTouchEvent(Event ev)`

该方法是ViewGroup的专有方法，未被重写的 `dispatchTouchEvent()` 方法内部将会调用此方法，判断是否需要拦截事件。

若返回值为 `true` 表示拦截事件，该事件不再传递给子控件， `dispatchTouchEvent()` 方法将会调用当前控件的 `onTouchEvent()` 方法处理事件。

若返回值为 `false` 表示不拦截事件，`dispatchTouchEvent()` 方法继续调用子控件的 `dispatchTouchEvent()` 方法，将事件传递给子控件处理。

🔷 `boolean onTouchEvent(Event ev)`

该方法是Activity、ViewGroup、View所共有的，用于对事件作出响应，未被重写时将会触发点击回调、长按回调、OnTouchListener接口回调。

若返回值为 `true` 表示消费事件，此时事件处理流程终止。

若返回值为 `false` 表示未消费事件，请求上层容器继续处理该事件。

<br />

我们通过最为常见的场景了解上述方法的作用：创建一个测试Activity，并在根布局中放置一个按钮，然后重写Activity、根布局、按钮的事件处理方法，不重写任何逻辑，仅添加日志输出语句。

`ACTION_UP` 事件的处理流程与 `ACTION_DOWN` 类似，此处我们以 `ACTION_DOWN` 事件为例进行分析：

1. 当手指接触到屏幕时， `ACTION_DOWN` 事件首先从Framework传递到Activity，Activity的 `dispatchTouchEvent()` 方法开始执行，该方法继续调用布局根容器的 `dispatchTouchEvent()` 方法，请求布局根容器处理该事件。
2. 布局根容器的 `dispatchTouchEvent()` 方法开始执行，首先调用自身的 `OnInterceptTouchEvent()` 方法检查是否需要拦截事件，默认策略为不拦截。接下来，根容器倒序遍历子View集合，查找触摸位置最顶层的子View（此处即按钮），然后调用它的 `dispatchTouchEvent()` 方法，请求子View处理该事件。
3. 按钮的 `dispatchTouchEvent()` 方法开始执行，该方法继续调用 `onTouchEvent()` 处理事件。 `onTouchEvent()` 方法将会回调开发者设置的点击事件监听器，并返回 `true` 消费事件，表明事件处理已经结束。
4. 按钮的 `onTouchEvent()` 方法返回值 `true` 逐步传递给根容器、Activity，整个处理流程结束。

此时down到达view时，未消费事件，因此ViewGroup收到该返回值后调用自身的OnTouchEvent，viewgroup默认不处理事件，OnTouchEvent也返回false，该返回值继续传递给Activity，执行activity的OnTouchEvent，整个流程结束。

引入事件序列的概念，一组事件的down在某个控件onTouchEvent被消费，即返回true， 那么ACTION_MOVE和ACTION_UP就会从上往下（通过dispatchTouchEvent）做事件分发往下传，就只会传到这个控件，不会继续往下传，如果ACTION_DOWN事件是在dispatchTouchEvent消费，那么事件到此为止停止传递，如果ACTION_DOWN事件是在onTouchEvent消费的，那么会把ACTION_MOVE或ACTION_UP事件传给该控件的onTouchEvent处理并结束传递。

虽然我们在view的OnTouchEvent保留了super.onTouchEvent(event)，只需等到up事件即可触发监听器，但down未消费返回给了viewgroup和activity，由activity作为最终处理者，虽然它什么都没做，后续up事件也不会再传递给内层控件，只会直接分发给activity。


此时我们可以归纳出三种界面组件的默认事件处理行为：

Activity

顶级容器，下发给DecorView的superDispatchTouchEvent方法，DecorView下发给contentView,此时就是我们通过layout设置的根容器

一般事件由子View消耗；如果子View未处理返回false，调用自己的 onTouchEvent。


ViewGroup

上层ViewGroup下发点击事件，调用ViewGroup的dispatchTouchEvent方法， dispatchTouchEvent内部调用 onInterceptTouchEvent方法，如果未拦截，遍历触摸位置找到最顶层的子View，调用dispatchTouchEvent方法。如果此时被拦截，事件不必下发给子元素，调用自己的 onTouchEvent。

ViewGroup收到触摸事件后，通常会递归调用触摸位置的子View的dispatchTouchEvent，交给子VIew处理，View作为末端，会调用其onTouchEvent方法处理事件，然后返回true消费事件，该返回值回传给ViewGroup、系统，处理流程结束。

ViewGroup中的方法实现包含查找子控件与分发的逻辑，如果我们重写该方法后只是简单地指定返回值，并未调用 `super.dispatchTouchEvent(ev)` ，也未自行实现相关逻辑，则控件无法执行任何功能。

ViewGroup中的方法实现为只拦截“鼠标拖动滚动条”事件，其他场景都不拦截事件，全部透传给子View处理。


View

上层ViewGroup下发点击事件，调用View的dispatchTouchEvent方法，View不能拥有子元素，因此没什么可分发的，继续调用自身的 `onTouchEvent` 方法

onTouchEvent返回true，表示处理完毕，流程结束；onTouchEvent返回false，表示事件未处理完，请求外层ViewGroup处理。



上面说的是可点击，可点击包括很多种情况，只要你给View注册了 onClickListener、onLongClickListener、OnContextClickListener 其中的任何一个监听器或者设置了 android:clickable=”true” 就代表这个 View 是可点击的。
另外，某些 View 默认就是可点击的，例如，Button，CheckBox 等。



单击事件(onClickListener) 需要两个两个事件(ACTION_DOWN 和 ACTION_UP )才能触发，如果先分配给onClick判断，等它判断完，用户手指已经离开屏幕，黄花菜都凉了，定然造成 View 无法响应其他事件，应该最后调用。(最后)
长按事件(onLongClickListener) 同理，也是需要长时间等待才能出结果，肯定不能排到前面，但因为不需要ACTION_UP，应该排在 onClick 前面。(onLongClickListener > onClickListener)
触摸事件(onTouchListener) 如果用户注册了触摸事件，说明用户要自己处理触摸事件了，这个应该排在最前面。(最前)
View自身处理(onTouchEvent) 提供了一种默认的处理方式，如果用户已经处理好了，也就不需要了，所以应该排在 onTouchListener 后面。(onTouchListener > onTouchEvent)
