## 内存信息
我们可以通过ActivityManager的 `getMemoryInfo(MemoryInfo info)` 方法获取设备内存总量、当前剩余内存等信息，当我们调用该方法时，它会将最新信息填入我们传入的MemoryInfo实例，然后我们可以通过其中的属性获取相关信息：

- `long totalMem` : 内存总量（单位：字节）。
- `long availMem` : 当前剩余内存（单位：字节）。
- `boolean lowMemory` : 当前剩余内存是否较低。如果该值为 `true` ，系统可能开始杀死后台进程以回收资源。
- `long threshold` : 剩余内存较低的阈值（单位：字节）。

# 系统兼容性
不同版本的Android提供的接口存在差异，部分接口在新旧版本的行为不同，部分用户可能仍使用旧版系统的设备，我们可以对系统版本进行判断，选择相应的业务代码实现功能。


"TestUICompatibility.java":

```java
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
    /* 系统版本高于Android 13 */
    Log.i(TAG, "系统版本高于Android 13，调用新的接口。");
    getPackageManager().getInstalledPackages(PackageManager.PackageInfoFlags.of(0L));
} else {
    /* 系统版本低于Android 13 */
    Log.i(TAG, "系统版本低于Android 13，调用旧的接口。");
    getPackageManager().getInstalledPackages(0);
}
```


各个系统版本的变更详情可在 [🔗 Android - 版本变更](https://developer.android.com/about/versions) 查询。

# 注解
## 简介
`androidx.annotation` 包提供了一些Android编码相关的注解，我们可以对代码元素进行标记和补充说明。

假设现有一个 `setBgColor(int color)` 方法，调用者必须阅读源码逻辑才能得知应当传入颜色数值还是颜色资源ID，如果我们为参数添加注解，将方法声明为 `setBgColor(@ColorInt int color)` ，调用者就知道此处应当传入色值而不是颜色资源的ID，如果误传ID编译器会给出警告。值得注意的是：注解不具有强制性，如果我们忽略警告对该接口传入ID也能编译通过，但运行时会把ID的数值作为颜色数值，导致显示效果与预期不符。


## 通用注解
以下注释用于对代码中的属性或方法进行补充说明，以便调用者正确地使用它们：

- `@Nullable` : 表示属性、参数或返回值可能为空值，我们在访问它们前应当先进行空值判断。
- `@NonNull` : 表示属性、参数或返回值不为空值，我们可以不进行空值判断。
- `@CallSuper` : 用于标记方法，子类重写该方法时，应当通过 `super()` 调用本方法。
- `@Keep` : 表示属性或方法不应当被混淆。
- `@RequiresPermission("<权限名称>")` : 表示访问该元素时应当具备相关权限。
- `@RequiresApi(<API Level>)` : 表示本元素仅在系统API版本大于等于该值时才能被访问。
- `@UiThread` : 表示本方法包含UI更新操作，应当在UI线程中执行。
- `@MainThread` : 表示本方法包含不涉及UI但必须在主线程中执行的操作，例如：发送Toast消息。
- `@WorkerThread` : 表示本方法包含耗时操作，应当在子线程中执行。
- `@BinderThread` : 表示本方法应当在Binder线程池中执行。

# 隐藏接口
Android SDK中的部分接口给系统组件使用的，不对普通应用开放，这些方法在源码中通常具有以下注解。

- `@hide` : 文档注解，表示变量或方法对普通应用不可见。
- `@SystemApi` : 方法注解，表示这是一个系统接口。
- `@PrivateApi` : 方法注解，表示这是一个隐藏接口。
- `@TestApi` : 方法注解，表示这是一个测试接口。
- `@UnsupportedAppUsage` : 属性、方法、类注解，表示该元素不对普通应用开放。

从AOSP编译生成SDK时，带有这些注解的元素将被删除，因此我们无法在代码中直接引用它们。

虽然隐藏方法在SDK中没有入口，但在运行时这些属性与方法是存在的，因此我们只要确保编译通过，以下方法调用隐藏接口：

1. 下载AOSP源码，移除隐藏方法相关注解，编译生成新的 `framework.jar` 并在当前工程中引用。
2. 将隐藏方法所在的源码文件复制到当前工程对应包中，确保编译通过。
3. 通过JVM的反射机制获取变量、方法等。

其中方法一只适合定制化的产品开发，因为每个工程只能导入一个`framework.jar`文件，且重新编译SDK操作比较繁琐。方法二只对独立的隐藏类有效，如果类中部分成员为Hide，另外部分成员非Hide，则无法实施，适用场景较少。方法三比较实用，能够适应不同的系统版本，且无需获取AOSP源码。

> 🚩 提示
>
> 上述方案只能使应用程序能够访问隐藏接口，但无法绕过接口本身的鉴权机制，应用程序必须具备接口所需的权限才能使用相关功能。

> 🚩 提示
>
> 在Android 10新增了对隐藏接口的调用限制，普通应用即使通过反射调用，也会出现NoSuchMethodException等异常，详见 [🧭 版本变更 - 变更一](#变更一) 。

## 变更一
### 摘要
自从Android 10 (API 29)开始，系统将会限制应用程序调用Android SDK中的部分隐藏接口，防止应用程序随意使用系统内部API。

各个系统版本接口受限情况可在 [🔗 Android - 非公开接口](https://developer.android.com/guide/app-compatibility/restrictions-non-sdk-interfaces) 页面查询。

### 详情
如果我们没有系统签名，通过反射调用部分隐藏接口时将得到NoSuchMethodException异常，并且会在Logcat中看到相关警告消息：

```text
Accessing hidden method Landroid/view/WindowManager;->shouldShowSystemDecors(I)Z (blocked,test-api, reflection, denied)
If this is a platform test consider enabling VMRuntime.ALLOW_TEST_API_ACCESS change id for this package.
```

### 兼容方案
当我们正在调试系统应用时，可以使用系统签名重新制作APK，或通过以下命令临时解除限制：

```shell
ADB# settings put global hidden_api_policy  1
```

调试完毕后可以通过以下命令恢复限制：

```shell
ADB# settings delete global hidden_api_policy
```

当我们正在编写库时，不便于直接添加系统签名，此时可以在测试应用中引入FreeReflection工具：

"build.gradle.kts":

```kotlin
dependencies {
    implementation("com.github.tiann:FreeReflection:3.1.0")
}
```

然后在自定义Application中重写 `attachBaseContext()` 方法，并调用 `Reflection.unseal()` 解除限制。

```text
@Override
protected void attachBaseContext(Context base) {
    super.attachBaseContext(base);
    // 解除Android 10及以上版本系统对隐藏接口的访问限制。
    Reflection.unseal(base);
}
```
