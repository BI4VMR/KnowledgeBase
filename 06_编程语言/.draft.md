隔离级别 - 等同于数据库ACID级别。

传播行为

是指多个业务方法都包括事务时，如何进行调度。

REQUIRE 没有事务则新建，已有事务则加入 （默认）

surrpots 没有事务则非事务执行，已有事务则加入

mandatory 必须在事务中运行，调用者没有开启事务则抛出异常

REQUIRE_new 挂起旧事务，在新事务中执行。挂起表示新旧事务无关，新事务回滚不影响旧的事务。

not_support 不在事务中运行，如果调用者线程有事务则将其挂起。

never 不准使用事务，调用者开启事务则抛出异常

nested 开启嵌套事务，新事务可以独立回滚，不影响外层事务；如果调用者没有事务则新建一个事务。

# Compose Multiplatform
## 简介
Compose Multiplatform是jetbrains维护的开源UI框架，最初叫Compose Desktop，旨在将android平台上的compose框架移植到桌面端JVM，实现UI代码的复用，后来又提供了ios/web等平台支持，因此更名为Multiplatform。


Skia/Skiko

它们的关系是一条自底向上的渲染技术栈：

 ┌─────────────────────────────────────────────┐
 │         Compose Multiplatform (UI 框架)       │
 │   ┌──────────────┐    ┌──────────────────┐  │
 │   │Compose Android│    │ Compose Desktop  │  │
 │   └──────┬───────┘    └───────┬──────────┘  │
 │          │                    │              │
 │          ▼                    ▼              │
 │   Android Canvas/        Skiko (Kotlin)      │
 │   RenderNode (HWUI)     Skia 的 Kotlin 绑定  │
 │          │                    │              │
 │          ▼                    ▼              │
 │      Skia (C++)           Skia (C++)         │
 │          │                    │              │
 │          ▼                    ▼              │
 │    OpenGL ES / Vulkan   OpenGL / Metal /     │
 │    (Android GPU)         Direct3D / Vulkan   │
 └─────────────────────────────────────────────┘

各层解释

1. Skia（C++ 库）

 - Google 开发的 2D 图形引擎，负责路径、文字、图片等所有 2D 绘制
 - Chrome、Android、Flutter、Firefox 都用它
 - 它本身不关心平台，只需一个 GPU backend（OpenGL/Vulkan/Metal/D3D）

2. Skiko（Skia for Kotlin）

 - JetBrains 做的 Skia 的 Kotlin/JVM 绑定层
 - 通过 JNI 调用 Skia C++ 代码
 - 打包了各平台（Windows/macOS/Linux）的 Skia 原生库
 - 提供 org.jetbrains.skia.* 包（Canvas、Image、Paint 等）

3. Compose Android vs Compose Desktop

┌───────────────┬───────────────────────────────────────────────────────────────┬────────────────────────────────────┐
│               │ Compose Android                                               │ Compose Desktop                    │
├───────────────┼───────────────────────────────────────────────────────────────┼────────────────────────────────────┤
│ 渲染后端      │ Android 自带的 HWUI/RenderNode（内部也用 Skia，但由系统管理） │ Skiko（JetBrains 自行管理的 Skia） │
├───────────────┼───────────────────────────────────────────────────────────────┼────────────────────────────────────┤
│ 窗口系统      │ Android WindowManager                                         │ AWT/Swing JFrame                   │
├───────────────┼───────────────────────────────────────────────────────────────┼────────────────────────────────────┤
│ GPU 调用      │ 系统走 OpenGL ES / Vulkan                                     │ Skiko 直接走 OpenGL / Metal / D3D  │
├───────────────┼───────────────────────────────────────────────────────────────┼────────────────────────────────────┤
│ Skia 版本     │ 跟随 Android 系统版本                                         │ 跟随 Skiko 发行版本                │
└───────────────┴───────────────────────────────────────────────────────────────┴────────────────────────────────────┘

关键区别

 - Compose Android 不直接使用 Skiko。Android 系统自带 Skia（编译在 libhwui.so 中），Compose Android 通过 android.graphics.Canvas → HWUI → Skia 这条路径渲染
 - Compose Desktop 没有 Android 系统支撑，所以 JetBrains 造了 Skiko 把 Skia 搬到 JVM 上，自己管理 OpenGL 上下文和窗口

一句话总结

 Skia 是共同的底层绘图引擎；Android 端由系统内置提供，Desktop 端由 JetBrains 通过 Skiko 桥接提供；Compose 是上层 UI 框架，两端共享声明式 API 但渲染管线不同。

# ffmpeg

要理解这一点，我们需要看看这两 waste 机制在底层是如何运作的：
1. FF_THREAD_FRAME (帧级多线程)

    工作原理： 它通过同时解码多个连续的帧来提高速度。例如，线程 1 解码第 1 帧，线程 2 解码第 2 帧。

    代价： 这种机制必然带来输入到输出的延迟（Latency）。因为要喂饱和管理多个线程，解码器通常需要缓存一定数量的帧（通常等于线程数）才能开始输出第一帧。

2. AV_CODEC_FLAG_LOW_DELAY (低延迟)

    工作原理： 强制解码器立刻输出当前解码的帧，绝对不进行内部缓存。它要求解码器满足“输入一帧，立刻输出一帧”的实时性要求，常见于实时视频通话或直播。

    代价： 无法利用需要跨帧缓存的优化算法，且无法进行帧级并行。

线程模式兼容 LOW_DELAY 吗？延迟表现适用场景FF_THREAD_FRAME❌ 不兼容较高（延迟 $N$ 帧）本地文件播放、视频转码（追求高吞吐量）FF_THREAD_SLICE兼容极低（0 帧延迟）实时连麦、低延迟直播（追求极致低延迟）

# log

SLF4J，即简单日志门面（Simple Logging Facade for Java），不是具体的日志解决方案，它只服务于各种各样的日志系统。按照官方的说法，SLF4J是一个用于日志系统的简单Facade，允许最终用户在部署其应用时使用其所希望的日志系统。

使用slf4j后，当我们在打印日志时，就可以使用下面的方式：

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
Logger logger = LoggerFactory.getLogger(Test.class);
logger.info("Hello World!")

    slf4j + logback： slf4j-api.jar + logback-classic.jar + logback-core.jar
    slf4j + log4j： slf4j-api.jar + slf4j-log412.jar + log4j.jar
    slf4j + jul： slf4j-api.jar + slf4j-jdk14.jar
    也可以只用slf4j无日志实现：slf4j-api.jar + slf4j-nop.jar

<dependency>
	<groupId>org.slf4j</groupId>
	<artifactId>slf4j-simple</artifactId>
	<version>1.7.25</version>
	<scope>compile</scope>
</dependency>


包名,角色,作用
slf4j-api,接口门面 (API),必选。 只包含日志接口（如 Logger.info()），没有具体实现。你的代码应该只依赖它。
