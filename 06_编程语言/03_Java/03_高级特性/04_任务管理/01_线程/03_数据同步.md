# 线程安全
## 竞态条件
线程的执行过程具有随机性，若同一变量被多个线程同时读写，可能导致逻辑错误，这种情况称为“竞态条件(Race Condition)”，它是破坏线程安全的主要原因。我们将被多个线程读写的变量称为“共享数据”，将访问同一变量的多线程代码称为“临界区(Critical Section)”。

操作系统和编译器无法分辨哪些线程之间涉及共享变量，因此只能保证单线程执行结果的正确性。在编码过程中，我们需要主动识别多线程并行可能导致的错误，并利用编程语言提供的工具协调线程，维护“线程安全”。

🔴 示例一：竞态条件。

在本示例中，我们以“多人购买同一款商品”场景为例，展示竞态条件可能造成的问题。

第一步，编写购买任务，模拟购买商品的过程。

"TestThreadSync.java":

```java
public class TestThreadSync {

    // 全局变量，表示商品库存数量。
    static int count = 10;

    static void example01() {
        // 定义任务：循环购买商品。
        Runnable task = () -> {
            while (true) {
                // 如果仍有存货，则进行购买。（动作一）
                if (count > 0) {
                    // 更新库存数量。（动作二）
                    count--;
                    // 输出日志。（动作三）
                    System.out.println(Thread.currentThread().getName() + " -> Buy one good, remain count is: " + count);
                } else {
                    break;
                }
            }
        };
    }
}
```

测试类中的全局变量 `count` 表示库存数量，所有线程都可以读写该变量。

购买任务将会循环检查库存数量，如果当前存在商品则购买1件，将库存数量减一；如果当前没有商品，则终止循环结束任务。

第二步，在测试代码中创建三个线程，模拟三个客户同时购买商品。

"TestThreadSync.java":

```java
// 定义三个线程，模拟三个客户。
Thread thread1 = new Thread(task);
thread1.setName("客户A");
Thread thread2 = new Thread(task);
thread2.setName("客户B");
Thread thread3 = new Thread(task);
thread3.setName("客户C");

// 所有客户开始购买商品。
thread1.start();
thread2.start();
thread3.start();
```

此时运行示例程序，并查看控制台输出信息：

```text
客户A -> Buy one good, remain count is: 9
客户B -> Buy one good, remain count is: 9

# 此处省略部分输出内容...

客户C -> Buy one good, remain count is: 0
客户B -> Buy one good, remain count is: -1
```

观察上述输出内容可以发现，在购买商品过程中，出现了以下两个问题：

🔷 重复购买

程序开始运行之后，“线程A”和“线程B”都在执行 `count--` 语句，将新的库存数量9写入CPU缓存；此时修改还没有从CPU缓存同步到内存中，另一个线程读取到的库存数量仍然为初始值10，因此它们输出的库存数量都为9，未能感知到对方的修改动作。

## 原子性
某些任务的多个步骤具有不可分割性，它们必须作为整体连续地执行，中途不能被其他线程干扰，我们将这种特性称为“原子性”。在任务运行期间，如果其他线程修改相关变量，可能导致任务本身出错；如果其他线程读取相关变量，此时变量值可能是任务的中间产物而非最终结果。

在前文示例中，由于我们未通过任何手段保证购买任务的原子性，当某个线程判断剩余商品为一件并暂停时，其他线程仍可进行购买，所以出现了最终商品数量为 `-1` 的逻辑错误。如果我们每次只允许单个线程执行操作，其他线程必须等待当前线程读写库存数量完成之后再进行购买，则保证了原子性，可以解决“商品数量超出范围”错误。

不满足原子性的典型场景如下文列表所示：

- `a = 1; b = 2;` ：线程可能在任意语句之间休眠，因此多条语句是非原子操作。
- `y = 2 * x;` ：该语句首先读取 `x` 的值、将该值扩大2倍再赋值给 `y` ，涉及两次外部变量交互，属于非原子操作。
- `i++` ： `i++` 实际上是“读取原值、将原值加1、写入新值”三个步骤，因此是非原子操作。
- `!state` ：与 `i++` 类似，具有“读取原值、将原值取反、写入新值”三个步骤。
- `Object o = new Object()` ：该语句需要经过“创建对象、执行构造方法、将对象引用赋值给变量 `o` ”三个步骤，因此是非原子操作。

## 可见性
对于多核CPU，通常各个核心拥有独立的L1和L2缓存，线程任务在单个核心上运行时，修改变量将会立刻更新缓存中的值，但内存速度比CPU缓存更慢，该变化稍后才会同步到内存中。由于缓存和内存同步存在时间差，一旦其他线程在同步间隙读取变量，就会得到未被更新的值，导致逻辑错误。

在前文示例中，某个线程购买商品后虽然更新了库存数量，但无法保证变化立即同步到内存中；此时其他线程看到的库存数量仍是旧值，导致多个线程重复购买同一件商品。如果我们使变量值在缓存和内存中保持同步，则保证了可见性，可以解决“同一商品被重复购买”错误。

不满足可见性的典型场景如下文列表所示：

- 共享标志位：通过一些线程更新标志位，并由另一些线程读取标志位。此场景虽然仅包含简单的读取和赋值操作，不存在原子性问题，但未满足可见性要求。
- 单例模式：当多个线程同时获取单例类实例时，我们应当确保首个线程新建的实例对其他线程可见，否则会出现逻辑错误。

## 有序性
处理器或编译器可能会调整代码的实际执行顺序，优先执行当前已具备执行条件的语句，这种行为被称为“指令重排序(Instruction Reorder)”。指令重排序遵循 `as-if-serial` 语义，不会改变具有依赖关系的语句顺序，因此在单线程环境下结果准确，但在多线程环境中可能造成混乱。

以下文代码片段为例：

```java
int a = 114 + 514;   // 语句1
int b = 1919 * 810;  // 语句2
int c = a * b;       // 语句3
```

由于“语句1”和“语句2”均为常量表达式，相互之间没有依赖关系，执行顺序可能是 `1 -> 2 -> 3` 或 `2 -> 1 -> 3` ，但不可能是 `1 -> 3 -> 2` 或 `3 -> 2 -> 1` 等，因为“语句3”依赖前两条语句的执行结果，必须在它们之后执行。

不满足有序性的典型场景如下文列表所示：

- 物理状态：有些标志位具有物理意义，例如：控制电灯点亮1秒再熄灭1秒，指令重排序会导致程序行为与预期不符。
- 单例模式：当我们创建对象时，“执行构造方法”和“为全局变量赋值”的顺序可能会颠倒，此时全局变量非空，但构造方法未执行，其他线程使用这种未初始化的单例会出现异常。


# 同步机制
## 简介
同步(Synchronize)是一种线程协调机制，可以限制同一时刻仅有单个线程执行临界区代码，其他线程必须排队等待，直到当前线程的临界区代码执行完毕，才能执行后续操作。

同步机制保证了原子性、可见性和有序性，因此实现了线程安全：

- 原子性：同一时刻只有一个线程执行临界区代码，即使该线程阻塞等待，其他线程也不能进入临界区；同步机制将多个线程串行化，保证了原子性。
- 可见性：前一个线程退出临界区时，缓存内容已经同步到内存，后续线程能够读取到最新的值，保证了可见性。
- 有序性：线程之间为串行关系，单个线程内部仍然可以重排序，但前一个线程的行为不影响后一个线程，保证了有序性。

虽然同步机制在一定程度上降低了程序的运行效率，但是能够保证多线程逻辑正确无误，因此是必要的。

## 同步代码块
同步代码块是一种自动锁定与解锁工具，其语法如下文代码块所示：

```text
synchronized (Object o) {
    // 临界区代码...
}
```

我们需要把临界区代码放置在同步代码块中，然后在 `synchronized` 关键字后的括号中指定同步对象。

🟠 示例二：同步代码块。

在本示例中，我们通过同步代码块解决前文示例中的竞态条件问题。

我们对线程任务进行改造，添加同步代码块，其他部分保持不变。

"TestThreadSync.java":

```java
Runnable task = () -> {
    while (true) {
        // 加锁，确保以下三个操作一次性执行完毕，中途不会被其他线程打断。
        synchronized (TestThreadSync.class) {
            if (count > 0) {
                count--;
                System.out.println(Thread.currentThread().getName() + " -> Buy one good, remain count is: " + count);
            } else {
                break;
            }
        }
    }
};
```

我们将三条临界区语句放置在同步代码块内部，并以当前类的Class对象作为同步对象，所有线程都需要竞争Class对象的锁，同时仅有单个线程能够连续地执行三条语句。

下文列表对一种可能发生的情况进行推演：

1. 三个线程同时运行至临界区，开始竞争同步对象的锁，“线程A”抢到了锁，开始执行临界区代码。
2. “线程B”与“线程C”未抢到锁，进入阻塞状态并暂停，不可被CPU调度。
3. “线程A”临界区代码执行完毕后，锁被自动释放；“线程B”与“线程C”竞争同步对象的锁。
4. “线程B”抢到了锁，开始执行临界区代码；“线程C”只能继续等待。
5. “线程B”临界区代码执行完毕后，锁被自动释放，“线程C”抢到了锁并执行临界区代码。

此时运行示例程序，并查看控制台输出信息：

```text
客户A -> Buy one good, remain count is: 9
客户B -> Buy one good, remain count is: 8

# 此处省略部分输出内容...

客户C -> Buy one good, remain count is: 1
客户A -> Buy one good, remain count is: 0
```

根据上述输出内容可知：

使用同步代码块保护三条语句后，重复购买与超出范围问题都未出现，符合业务要求。

## 同步方法
若方法中的所有语句都属于临界区，我们可以直接在方法签名处添加 `synchronized` 关键字，相比同步代码块更加简洁。

🟡 示例三：同步方法。

在本示例中，我们通过同步方法解决前文示例中的竞态条件问题。

第一步，对线程任务进行改造，将临界区代码提取为方法，并在方法签名处添加 `synchronized` 关键字。

"TestThreadSync.java":

```java
// 同步方法，整个方法体都是临界区。
synchronized static boolean buy() {
    if (count <= 0) {
        // 库存售空，返回 false。
        return false;
    } else {
        // 库存非空，购买一件商品并返回 true。
        count--;
        System.out.println(Thread.currentThread().getName() + " -> Buy one good, remain count is: " + count);
        return true;
    }
}
```

第二步，在线程任务中循环调用 `buy()` 方法，直到该方法返回商品售空状态时结束任务。

"TestThreadSync.java":

```java
// 定义任务：循环购买商品。
Runnable task = () -> {
    while (buy()) {
        // 循环购买，直到库存售空。
    }
};
```

同步方法无法手动指定同步对象，在静态方法中以当前类的Class对象作为同步对象，上述示例中等价于 `synchronized (TestThreadSync.class) {}` ；在实例方法中以当前对象作为同步对象，等价于 `synchronized (this) {}` 。

## 同步对象
每个Java对象都有一把“锁”（本质上是JVM中记录的一系列状态变量），当多个线程执行关联到同一对象的同步代码块时，它们需要竞争对象的锁，只有抢夺到锁的线程可以进入临界区，其他线程则阻塞并等待前一个线程退出临界区。

关联到相同对象的线程必须排队等待、依次执行；而关联到不同对象的线程则互不干扰。在实际应用中，我们应当为各个业务分配合适的同步对象，避免不必要的等待，提高程序的并发效率。

🟢 示例四：选择合适的同步对象。

在本示例中，我们展示同步对象选择不当所造成的影响，并提供合理的解决方案。

下文代码块展示了一种不恰当的同步机制用法：

```java
class BusinessManager {

    private int sum = 0;

    private final List<EventListener> listeners = new ArrayList<>();

    // 注册监听器
    public synchronized void addListener(EventListener l) {
        if (!listeners.contains(l)) {
            listeners.add(l);
        }
    }

    // 注销监听器
    public void removeListener(EventListener l) {
        listeners.remove(l);
    }

    // 计数业务
    public synchronized void counting() {
        sum++;
        System.out.println("New count is: " + sum);
    }
}
```

该类具有“监听器管理”和“计数”两个互不相关的业务，但它们都被声明为同步方法，此时同步对象均为BusinessManager实例，多个线程调用 `addListener()` 和 `counting()` 方法时需要竞争BusinessManager实例的锁， `counting()` 方法需要等待 `addListener()` 方法操作完毕后才能执行，这显然是不合理的。

除此之外， `removeListener()` 方法未启用同步机制，此时该方法不受任何并行限制，在 `addListener()` 方法执行过程中其他线程仍然可以调用 `removeListener()` 方法，可能导致逻辑错误。

我们对示例代码进行修改，解决前文出现的两个问题：

```java
class BusinessManager2 {

    private int sum = 0;

    // 创建对象，作为计数业务的锁。
    private final Object lock = new Object();

    private final List<EventListener> listeners = new ArrayList<>();

    // 注册监听器
    public void addListener(EventListener l) {
        synchronized (listeners) {
            if (!listeners.contains(l)) {
                listeners.add(l);
            }
        }
    }

    // 注销监听器
    public void removeListener(EventListener l) {
        synchronized (listeners) {
            listeners.remove(l);
        }
    }

    // 计数业务
    public void counting() {
        synchronized (lock) {
            sum++;
            System.out.println("New count is: " + sum);
        }
    }
}
```

`addListener()` 和 `removeListener()` 方法属于“监听器管理”业务，相关调用应当串行执行，需要关联到同一同步对象，我们将监听器集合作为同步对象即可。

`counting()` 方法属于“计数”业务，相关调用应当串行执行，但此处业务变量 `count` 是基本数据类型，无法作为同步对象，因此我们声明一个Object类型的变量 `lock` 作为同步对象。

此时两个业务各自关联到专属的同步对象，方法调用互不干扰，并发效率得到了提高。

## 等待与唤醒
在前文示例中，购买商品场景只会单次运行，商品售空即整个程序结束。若我们希望实现“商家持续补货、客户持续购买”的场景，就需要在商品售空时让客户线程暂停，待商家补货后再恢复客户线程的执行。

对于这种需求，忙等待或定时休眠均存在弊端：

- 轮询：客户线程在循环中不断地尝试购买商品。这种方式确实实现了抢占，但无货时间客户线程仍然在空转，CPU无法执行其他任务，浪费系统资源。

- 定时检查库存：我们可以在客户线程每次轮询结束后插入 `sleep()` 语句，使其等待一段时间再尝试购买。这种方式能够降低CPU占用率，但商家补货时机是不确定的，客户线程无法在补货后立刻执行购买。


更好的方案是利用线程的等待与唤醒机制：当商品无货时，客户线程主动释放锁并进入等待状态，CPU不会调度它们，不浪费系统资源；当商家线程补货完成后，主动唤醒所有等待中的客户线程，让它们重新竞争锁并执行购买逻辑。

Object类提供了以下方法用于实现线程间的等待与唤醒：

- `void wait()` : 

使当前线程释放持有的锁并进入等待状态，直到其他线程调用同一对象的 `notify()` 或 `notifyAll()` 方法将其唤醒，才可重新竞争锁并恢复运行。

由于线程在等待状态可能被外部中断，该方法会抛出受检异常 `InterruptedException` ，需要显式处理。

- `void wait(long timeoutMillis)` : 指定一个最大等待时长，如果休眠直到该时长还没收到notify，就会自动唤醒，在某些场景下可以防止死锁。

与 `wait()` 类似，但超过指定的等待时长后，若仍未收到唤醒通知，线程会自动恢复运行，可在某些场景下防止死锁。

- `notify()` : 当前临界区代码执行完毕后，唤醒任意一个正在等待同步对象锁的线程。该方法的调用者必须持有同步对象的锁，即在同步代码块中进行操作。

- `void notifyAll()` : 当前临界区代码执行完毕后，唤醒所有正在等待当前对象锁的线程，使它们竞争锁。`notify()` 和 `notifyAll()` 是非公平锁，具体实现由系统决定，PC端的JVM中通常以wait顺序而定，先进先出，并非真正的随机。

上述方法必须在同步代码块或同步方法中调用，即调用线程必须先持有对应对象的锁，否则将抛出 `IllegalMonitorStateException` 异常。

🔴 示例五：等待与唤醒机制。

在本示例中，我们以“商家间隔补货、客户持续抢购”场景为例，展示线程的等待与唤醒机制。

第一步，声明全局变量和锁对象。

"TestWaitNotify.java":

```java
// 全局变量，表示商品库存数量。
static int count = 0;

// 锁对象，生产者和消费者线程都需要通过它进行同步。
final static Object lock = new Object();

public static void main(String[] args) {
    // 生产者任务（模拟进货的过程）
    Runnable producerTask = () -> {
        while (true) {
            // 每次补货之前休眠3秒，模拟进货的间隔。
            Thread.sleep(3000L);
    
            synchronized (lock) {
                // 补货
                count++;
                System.out.println(Thread.currentThread().getName() + " -> 已补货1件，当前库存: " + count);
                // 唤醒所有等待的消费者线程，让它们开始抢购。
                lock.notifyAll();
            }
        }
    };
}
```

生产者线程首先休眠3秒模拟进货间隔，然后进入同步代码块，将库存数量加1，最后调用 `notifyAll()` 方法唤醒所有等待中的消费者线程。

# "volatile"关键字
## 简介
`volatile` 关键字用于修饰共享变量，表示这些变量是“易变的”，要求系统在CPU缓存与内存之间同步它们的值，并禁止指令重排序。

变量同步是平台相关的能力，以 `x86` 平台为例：CPU更新被 `volatile` 修饰的变量时将额外添加 `LOCK` 指令前缀，使得内存中的值也被同步更新；其他CPU核心能够通过MESI协议感知到变化，并将自身缓存的值标记为失效，下次访问时直接从内存中读取最新的值。

## 变量赋值场景
`volatile` 机制是无锁的，相比同步机制效率更高。对于简单的变量赋值场景，其本身就是原子操作，我们仅需通过 `volatile` 机制确保可见性与有序性即可。

`volatile` 的典型应用场景是线程中断标志位：

"Thread.java":

```java
class Thread {

    // 中断标志位
    private volatile boolean interrupted;

    // 请求中断方法
    public void interrupt() {
        // 此处已省略部分代码...

        interrupted = true;
    }
}
```

Thread实例的 `interrupt()` 方法可能被任意线程调用，为了确保该方法对变量 `interrupted` 的修改第一时间同步到线程内部，源码将 `interrupted` 变量标记为 `volatile` 。

在线程内部，我们可以使用该变量判断是否需要终止任务：

```java
new Thread(() -> {
    // 循环执行任务，如果发现中断请求则结束任务。
    while (!Thread.currentThread().isInterrupted()) {
        // 任务语句
    }
});
```

`volatile` 机制不适用于“先读后写”等场景，例如 `!interrupted` 或 `i++` 等语句，这些语句实际上包含多个步骤，而  `volatile` 无法保证原子性。

## 单例模式场景
`volatile` 经常出现在DCL单例模式中，示例代码详见下文代码块：

```java
public class LazyDCLSingleton {

    // 使用"volatile"关键字禁止指令重排序，防止出现异常。
    private volatile static LazyDCLSingleton instance = null;

    // 对外公开的方法，供外界获取当前类的实例。
    public static LazyDCLSingleton getInstance(int arg1) {
        // 仅在实例未创建时进入同步区域
        if (instance == null) {
            synchronized (LazyDCLSingleton.class) {
                // 二次判断，防止当前线程通过判断后休眠，此时其他线程成功创建实例，当前线程再恢复导致异常。
                if (instance == null) {
                    instance = new LazyDCLSingleton(arg1);
                }
            }
        }
        return instance;
    }
}
```

在 `getInstance()` 方法中，我们使用同步机制确保仅有单个线程创建实例，但第一条 `if (instance == null)` 语句在临界区外部，不受同步代码块限制；若实例创建后未执行构造方法就被赋值给 `instance` 变量，其他线程可通过 `if (instance == null)` 判断并获取该实例，导致未初始化错误。

当我们为 `instance` 变量添加 `violate` 关键字后，创建实例、执行构造方法、为变量赋值三个操作必须按顺序执行，可以解决上述问题。

# CAS
使用锁时，线程获取锁是一种悲观锁策略，即假设每一次执行临界区代码都会产生冲突，所以当前线程获取到锁的时候同时也会阻塞其他线程获取该锁。而CAS操作（又称为无锁操作）是一种乐观锁策略，它假设所有线程访问共享资源的时候不会出现冲突，既然不会出现冲突自然而然就不会阻塞其他线程的操作。因此，线程就不会出现阻塞停顿的状态。那么，如果出现冲突了怎么办？无锁操作是使用CAS(compare and swap)又叫做比较交换来鉴别线程是否出现冲突，出现冲突就重试当前操作直到没有冲突为止。



CAS比较交换的过程可以通俗的理解为CAS(V,O,N)，包含三个值分别为：V 内存地址存放的实际值；O 预期的值（旧值）；N 更新的新值。当V和O相同时，也就是说旧值和内存中实际的值相同表明该值没有被其他线程更改过，即该旧值O就是目前来说最新的值了，自然而然可以将新值N赋值给V。反之，V和O不相同，表明该值已经被其他线程改过了则该旧值O不是最新版本的值了，所以不能将新值N赋给V，返回V即可。当多个线程使用CAS操作一个变量是，只有一个线程会成功，并成功更新，其余会失败。失败的线程会重新尝试，当然也可以选择挂起线程

CAS的实现需要硬件指令集的支撑，在JDK1.5后虚拟机才可以使用处理器提供的CMPXCHG指令实现。

3.1.4 CAS的问题

1. ABA问题 因为CAS会检查旧值有没有变化，这里存在这样一个有意思的问题。比如一个旧值A变为了成B，然后再变成A，刚好在做CAS时检查发现旧值并没有变化依然为A，但是实际上的确发生了变化。解决方案可以沿袭数据库中常用的乐观锁方式，添加一个版本号可以解决。原来的变化路径A->B->A就变成了1A->2B->3C。java这么优秀的语言，当然在java 1.5后的atomic包中提供了AtomicStampedReference来解决ABA问题，解决思路就是这样的。

2. 自旋时间过长

使用CAS时非阻塞同步，也就是说不会将线程挂起，会自旋（无非就是一个死循环）进行下一次尝试，如果这里自旋时间过长对性能是很大的消耗。如果JVM能支持处理器提供的pause指令，那么在效率上会有一定的提升。

3. 只能保证一个共享变量的原子操作

当对一个共享变量执行操作时CAS能保证其原子性，如果对多个共享变量进行操作,CAS就不能保证其原子性。有一个解决方案是利用对象整合多个共享变量，即一个类中的成员变量就是这几个共享变量。然后将这个对象做CAS操作就可以保证其原子性。atomic中提供了AtomicReference来保证引用对象之间的原子性。



变量计算前保留一份旧值a，计算完成后结果值为b，把b刷到内存之前先比较a是否和内存中变量一致，如果一致，就把内存中的变量赋值为b，不一样，重新获取内存中变量值，重复一遍操作，一直到a和内存中一致，操作结束。
