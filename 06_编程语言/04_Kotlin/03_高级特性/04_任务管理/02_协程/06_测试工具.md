# 简介
`kotlinx-coroutines-test` 模块提供了一些实用的协程测试工具，我们可以通过以下Gradle配置将其引入到工程中，然后按需选用相关工具。

```groovy
dependencies {
    testImplementation 'org.jetbrains.kotlinx:kotlinx-coroutines-test:1.8.1'
}
```

上述内容也可以使用Kotlin语言编写：

"build.gradle.kts":

```kotlin
dependencies {
    testImplementation("org.jetbrains.kotlinx:kotlinx-coroutines-test:1.8.1")
}
```


# 基本应用
为了执行被测对象的挂起函数，我们首先需要在测试代码中构建一个协程环境；由于测试线程与业务协程是分别运行的，我们还要确保测试线程等待协程运行完毕后再退出。

`runBlocking {}` 方法符合上述的两项要求，该方法能够提供协程环境，并且阻塞测试线程直到内部的语句块执行完毕。 `kotlinx-coroutines-test` 模块提供了 `runTest {}` 方法，它的行为与 `runBlocking {}` 方法基本一致，并且新增了一些适合在测试环境中使用的特性。

`runTest {}` 是TestScope的扩展方法，它会创建一个TestScope实例作为协程环境，我们可以在此处调用挂起函数。TestScope将会跳过挂起函数中的 `delay()` 延时，以减少测试用例的运行时长。

🔴 示例一：测试挂起函数。

在本示例中，我们通过 `runTest()` 方法创建TestScope，并在该作用域内调用挂起函数。

```kotlin
/* 被测方法 */
suspend fun task() {
    println("Task start. Time:[${getTime()}]")
    delay(60_000L)
    println("Task end. Time:[${getTime()}]")
}


/* 测试代码 */
@Test
fun test_Base() = runTest {
    task()
    println("Test end.")
}
```

此时运行示例程序，并查看控制台输出信息：

```text
Task start. Time:[09:56:25.048]
Task end. Time:[09:56:25.054]
Test end.
```

根据上述输出内容可知：

我们在挂起函数中设置了一分钟的延时，但实际上该方法耗时为6毫秒，没有真的等待一分钟。


# 控制协程进度
TestScope提供了两个变量以便我们控制协程的执行过程：

- `testScheduler` : 测试调度器。该调度器可以控制协程的执行进度，我们需要创建测试作用域并注入到被测实例中，然后在适当的时机推进任务进度，确保业务协程执行完毕再检查结果。
- `backgroundScope` : 后台任务作用域。该作用域中的协程只会静默执行，不受TestScope推进任务请求的影响，遇到错误也不会阻断测试，其中的协程将在测试用例结束后自动取消。如果被测对象中的部分协程不影响测试结果，可以通过该作用域执行。

TestScope的 `advanceUntilIdle()` 或 `advanceTimeBy(ms: Long)` 方法都可以挂起测试协程并推进业务协程任务进度，前者将会执行所有任务直到完成，后者只会推进指定的 `delay()` 时长。

🟠 示例二：通过TestScheduler控制协程进度。

在本示例中，我们使用TestScheduler控制被测协程的执行进度，确保测试协程等待被测协程执行完毕。

第一步，编写业务代码。

"DataManager.kt":

```kotlin
class DataManager(
    dispatcher: CoroutineDispatcher = Dispatchers.IO
) {
    private val scope = CoroutineScope(dispatcher)

    var freeSpace: Int = 0
        private set

    fun clearCache() {
        scope.launch {
            // 模拟耗时操作...
            delay(1000L)

            // 清理完毕，增加可用空间。
            freeSpace = 100
            println("Clean completed in coroutine task.")
        }
    }
}
```

DataManager的 `freeSpace` 属性用于指示当前可用的缓存空间，业务方法 `clearCache()` 将在协程中清理缓存，并于任务完成时更新 `freeSpace` 属性。

第二步，编写测试代码。

"TestTools.kt":

```kotlin
@Test
fun test_TestScheduler() = runTest {
    // 使用TestScope提供的Scheduler，创建测试调度器。
    val dispatcher = StandardTestDispatcher(testScheduler)

    // 将测试调度器注入到待测组件，并执行业务方法。
    val manager = DataManager(dispatcher)
    println("Free space is ${manager.freeSpace} before clean.")
    manager.clearCache()

    // 测试协程等待业务协程执行完成
    advanceUntilIdle()

    // 业务协程执行完毕后再断言结果
    println("Free space is ${manager.freeSpace} after clean.")
    Assert.assertTrue(manager.freeSpace > 10)
}
```

我们首先通过TestScope提供的TestScheduler创建Dispatcher，然后将其注入到被测对象中，使被测对象的协程可以被TestScheduler控制。

当我们调用 `clearCache()` 方法开启业务协程后，又在测试协程中调用了 `advanceUntilIdle()` 方法，此时测试协程将挂起并等待业务协程执行完毕；当业务协程执行完毕后，测试协程从挂起状态恢复并执行断言。

我们首先移除 `advanceUntilIdle()` 语句，运行示例程序，并查看控制台输出信息：

```text
Free space is 0 before clean.
Free space is 0 after clean.
Clean completed in coroutine task.

java.lang.AssertionError
	at org.junit.Assert.fail(Assert.java:87)
```

由于测试协程并未等待被测方法 `clearCache()` 中的业务协程执行完毕，因此其读取到的 `freeSpace` 变量仍为初始值 `0` ，结果与预期不符。

然后我们添加 `advanceUntilIdle()` 语句，再次运行示例程序，并查看控制台输出信息：

```text
Free space is 0 before clean.
Clean completed in coroutine task.
Free space is 100 after clean.
```

此时测试协程中的断言语句在 `clearCache()` 协程执行完毕之后读取 `freeSpace` 变量并进行断言，结果符合预期。

---

> 🚩 提示
>
> 只有关联到TestScheduler的协程才会受到 `advanceUntilIdle()` 等方法控制，因此要求被测代码按照前文章节 [🧭 协程环境 - 编码风格](./04_协程环境.md#编码风格) 中的推荐写法进行组织，以便我们在测试环境中注入测试调度器。
>
> 若被测代码直接引用内置调度器创建协程环境，我们无法通过 `advanceUntilIdle()` 控制其任务进度，此时只能预估任务耗时并使测试线程休眠更久的时长来实现等待。
>
> ```kotlin
> /* 业务代码 */
> fun foo() {
>     CoroutineScope(Dispatchers.IO).launch {
>         // 模拟耗时操作...
>         delay(1000L)
>     }
> }
> 
> 
> /* 测试代码 */
> @Test
> fun test() {
>     foo()
> 
>     // 测试线程休眠比业务协程略长的一段时间，等待业务协程执行完毕。
>     Thread.sleep(1100L)
> 
>     // 此时可以执行断言等操作
> }
> ```
>
