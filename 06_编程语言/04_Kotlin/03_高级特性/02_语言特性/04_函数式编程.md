# 简介
面向对象模型明确了实体之间的交互关系，能够更好地组织大型项目的结构，但在部分场景中反而提高了复杂度。例如：我们希望将列表中的元素倒序排列，需要创建Comparator接口的匿名对象，并实现 `compare()` 抽象方法设置排序规则。

🔴 示例一：将整数列表中的元素倒序排列。

在本示例中，我们定义一个整数列表，并将其中的元素倒序排列。

"TestBase.kt":

```kotlin
// 创建测试列表
val list = mutableListOf(1, 2, 3, 4, 5)

// 设置排序规则，将所有元素降序排列。
list.sortWith(object : Comparator<Int> {

    override fun compare(o1: Int, o2: Int): Int {
        // 返回当前两个数值的比较结果
        return o2.compareTo(o1)
    }
})
```

我们只想将“倒序排列”这一行为传递给 `sort()` 方法，按照面向对象的语法风格，我们需要创建Comparator接口的匿名对象作为参数，语法较为繁琐。

Comparator接口仅有唯一的抽象方法 `compare()` ，我们创建接口的匿名对象就是为了重写抽象方法传递行为，此类接口被称为函数式接口(Functional Interface)。

函数式编程的思想旨在将行为直接作为参数在方法间传递，无需创建匿名对象，适合表达数据过滤、排序、转换等场景的逻辑，相比面向对象语法更加简洁。


# Lambda表达式
## 简介
Lambda表达式是一种匿名函数，我们可以将函数式接口的匿名对象替换为等价的Lambda表达式，简化代码。

以Comparator接口为例，我们对前文“示例一”进行修改，利用Lambda表达式设置排序规则。

🟠 示例二：将整数列表中的元素倒序排列（Lambda表达式）。

在本示例中，我们定义一个整数列表，并将其中的元素倒序排列。

"TestBase.kt":

```kotlin
// 创建测试列表
val list = mutableListOf(1, 2, 3, 4, 5)


// 设置排序规则，将所有元素降序排列。
list.sortWith({ o1, o2 -> o2.compareTo(o1) })

// Lambda优化：如果Lambda表达式是方法的最后一个参数，可以将其移到括号外面。
list.sortWith { o1, o2 -> o2.compareTo(o1) }
```

此处的Lambda表达式 `{ o1, o2 -> o2.compareTo(o1) }` 等价于前文“示例一”中的匿名对象，语法更加简洁。

## 语法
Lambda表达式的标准格式为：

```text
{<参数1名称>: <参数1类型>, <参数2名称>: <参数2类型>, ... -> <函数体>}
```

在Kotlin语言中，无法以 `return` 关键字指示Lambda表达式的结果，我们约定将函数体末尾语句的值作为整个表达式的结果，例如：在 `{ o1, o2 -> o2.compareTo(o1) }` 中函数体只有 `o2.compareTo(o1)` 一条语句，则该语句的值就是整个表达式的结果。

若要在末尾语句之前返回结果，我们可以使用 `retrun@<标签>` 语法，例如：

```kotlin
list.sortWith { o1, o2 ->
    // 特殊规则，数值 `0` 排在最前面。
    if (o1 == 0) return@sortWith -1
    if (o2 == 0) return@sortWith 1

    o2.compareTo(o1)
}
```

前文“示例一”中的 `compare(o1: Int, o2: Int): Int` 方法转换为Lambda表达式后记作 `{ o1: Int, o2: Int -> o2.compareTo(o1) }` ，该表达式还可以进一步简化。

该表达式由函数式接口转换而来，编译器能够根据抽象方法的参数列表推断参数类型，因此我们可以省略参数类型，记作： `{ o1, o2 -> o2.compareTo(o1) }` 。

除了上述示例中出现的情况，Lambda表达式还有其他简化规则。

当Lambda表达式仅有单个参数时，其默认名称为 `it` ，我们可以省略参数声明，例如 `{ text -> println(text) }` 等价于 `{ println(it) }` 。

当Lambda表达式的函数体并未使用某个参数时，我们可以将参数名称改为下划线( `_` )形式，例如： `{ text -> println("示例文本") }` 等价于 `{ _ -> println("示例文本") }` 。

若Lambda表达式仅调用单个方法，且该方法的参数列表、返回值类型与表达式完全相同，整个表达式可以简化为方法引用。

🟡 示例三：方法引用。

在本示例中，我们使用方法引用简化Lambda表达式。

第一步，将排序规则提取为一个方法。

"TestBase.kt":

```kotlin
fun reverse(a: Int, b: Int): Int {
    return b.compareTo(a)
}
```

第二步，创建测试数据并设置排序规则。

"TestBase.kt":

```kotlin
// 创建测试列表
val list = mutableListOf(1, 2, 3, 4, 5)

// 将Lambda参数传递给 `reverse()` 方法，并返回其结果。
list.sortWith { o1, o2 -> reverse(o1, o2) }

// 此时可以简化为方法引用形式： `::<方法名称>` 。
list.sortWith(::reverse)
```

在上述代码中，Lambda表达式 `{ o1, o2 -> reverse(o1, o2) }` 的参数列表、返回值类型与 `reverse()` 方法一致，因此我们可以将整个表达式替换为 `::reverse` 。

该示例中的 `reverse()` 是文件级方法，故方法引用格式为 `::<方法名称>` ；如果需要引用实例方法，则语法为 `<对象名称>::<方法名称>` 或 `this::<方法名称>` 。

---

当我们在Kotlin代码中调用Java方法时，函数式接口参数将被转换为对应的Lambda表达式参数，以便我们通过Lambda表达式传递行为，例如： `int sort(Comparator<? super E> c)` 方法对应 `fun sort { c: (Int, Int) -> Int }` 方法。

当我们在Kotlin代码中调用Kotlin方法时，即使方法具有函数式接口参数，上述转换规则也不会生效，这是因为Kotlin语言提供了 [🧭 高阶函数](#高阶函数) 特性，允许我们直接将函数作为参数传递。例如： `fun setListener(callback: Consumer<T>)` 方法不会生成对应的 `fun setListener { param: T -> Unit }` 方法，我们应当将方法定义为 `fun setListener(onCall: (param: T) -> Unit)` 。

---

Lambda表达式不仅是函数式接口匿名对象的等价语法，在运行时二者的行为也存在差异。

如果Lambda表达式未引用任何外部变量，编译器只会创建单例对象，每次表达式被执行时均复用该对象，例如： `{ text -> println(text) }` 。函数式接口每次被执行前，都会创建一个匿名对象，在此场景中内存占用更高。

如果Lambda表达式引用了外部变量，每次表达式被执行前都要创建一个对象保存变量引用，例如： `instance::method` ，此时Lambda表达式的性能与函数式接口匿名对象一致。

## 引用表达式
有时我们希望将Lambda表达式保存在变量中，以便多次复用。Kotlin语言支持将函数作为变量类型，我们可以直接声明变量保存对表达式的引用。

🟢 示例四：引用Lambda表达式。

在本示例中，我们使用Lambda表达式声明线程任务，并创建多个线程执行该任务。

"TestBase.kt":

```kotlin
// 将Lambda表达式的引用保存在变量中
val task: () -> Unit = { println("Hello, World!") }

// 创建线程，复用Lambda表达式。
Thread(task).start()
Thread(task).start()
```

## "this"关键字
当我们使用匿名内部类或Lambda表达式实现函数式接口时，都可以使用 `this` 关键字，但两种场景中该关键字的含义有所差异。

🔵 示例五： `this` 关键字。

在本示例中，我们辨析Lambda表达式与匿名内部类中 `this` 关键字的区别。

第一步，在测试类中编写一个成员方法，使用Lambda形式和匿名类形式分别创建Runnable接口的实例。

"TestBase.kt":

```kotlin
class Test() {

    fun testThis() {
        // 测试类实例的 `this` 引用
        println("this in test class: $this")


        // Lambda表达式中的 `this` 指向包含它的实例，即 `TestBase` 的实例。
        val lambda: () -> Unit = { println("this in lambda: $this") }
        lambda.invoke()


        // 匿名内部类中的 `this` 指向类的实例，可以访问其属性。
        val object1: Runnable = object : Runnable {

            // 匿名内部类可以拥有属性
            private val name = "Runnable"

            override fun run() {
                println("this in annonymous class: $this")
                // `this` 可以访问匿名内部类的属性
                println("get name by this: ${this.name}")
            }
        }
        object1.run()
    }
}
```

第二步，在 `main()` 方法中创建测试类的实例，并调用该成员方法。

此时运行示例程序，并查看控制台输出信息：

```text
this in test class: net.bi4vmr.study.lambda.Test@7e9e5f8a
this in lambda: net.bi4vmr.study.lambda.Test@7e9e5f8a
this in annonymous class: net.bi4vmr.study.lambda.Test$testThis$object1$1@4554617c
get name by this: Runnable
```

根据上述输出内容可知：

匿名内部类的 `this` 指向自身实例，可以访问匿名内部类的成员变量；Lambda表达式的 `this` 等价于外部类的 `this` 。

虽然Lambda表达式在JVM内部仍有实例，但函数是无状态的，不能拥有成员变量，若将 `this` 指向Lambda实例没有实际意义；因此Lambda表达式中的 `this` 仍指向外部类，不具有独立的上下文，符合函数的定义。

## 捕获外部变量
方法中的局部变量将在方法执行完毕后立即销毁，但Lambda表达式可能在方法调用结束后才开始异步执行，为了使Lambda表达式在外部方法执行完毕后仍能访问局部变量，编译器需要进行额外的隐式处理，这种情况被称为变量捕获。

🟣 示例六：捕获外部变量。

在本示例中，我们尝试在Lambda表达式中访问与修改外部变量。

"TestBase.java":

```java
class Test2 {

    fun testCapture() {
        // 定义一个局部变量
        var num = 0

        // 定义Lambda表达式，尝试访问和修改外部变量。
        val lambda: () -> Unit = {
            // 访问外部变量
            println("Read `num` in lambda:[$num]")

            // 修改外部变量
            num++
        }

        // 执行Lambda表达式
        lambda()

        // 在Lambda表达式外部访问修改后的变量值
        println("Read `num` outside lambda:[$num]")
    }
}
```

此时运行示例程序，并查看控制台输出信息与界面外观：

```text
Read `num` in lambda:[0]
Read `num` outside lambda:[1]
```

编译器会将Lambda表达式所引用的外部变量复制到Lambda对象中，例如上述代码中的局部变量 `s` ，即使 `testCapture()` 方法执行完毕变量被销毁，Lambda对象仍持有 `s` 指向的内存地址，能够访问原变量的值。

与Java不同，Kotlin允许我们在Lambda表达式中修改外部变量的值，因为Kotlin编译器会将原始变量替换为IntRef等容器，修改变量值本质上是修改容器中的内容，容器本身并未被重新赋值，这种机制确保了Lambda内外的修改行为实时同步。
